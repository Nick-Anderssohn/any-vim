# Phase 6 — UI Design Contract

> Visual and interaction contract for Phase 6: Polish and Configuration.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Platform note:** AnyVim is a native macOS AppKit application. There is no web design
> system, no CSS, and no component library. All typography, color, and spacing are governed
> by macOS system APIs. This UI-SPEC describes the AppKit-native visual contract only.

## Spacing Scale

AppKit NSMenu spacing is controlled by the OS. Explicit spacing tokens are not applicable
to this phase's surface area (menu bar icon + menu items).

The only developer-controlled spacing is the NSOpenPanel configuration — this uses
AppKit defaults (no custom layout needed).

| Token | Value | Usage |
|-------|-------|-------|
| Separator | NSMenuItem.separator() | Between permission/hotkey status group and vim path group; between vim path group and login/quit group |

Exceptions: No pixel-level spacing tokens. AppKit handles all NSMenu item padding and
layout per the macOS Human Interface Guidelines automatically.

---

## Typography

AppKit renders all NSMenuItem titles in the system font at the system-standard size.
No custom font sizes or weights are set. This is correct behavior for a menu bar app —
overriding system menu typography would violate macOS HIG.

| Role | Value | Usage |
|------|-------|-------|
| Menu item title | System font, regular weight, macOS-default size (~13pt) | All NSMenuItem titles |
| Disabled info item | System font, regular weight, dimmed via `isEnabled = false` | "Vim: /path/to/vim" display item |

Typography is not configurable in this phase. No custom fonts, sizes, or weights are
introduced.

---

## Color

All color in this phase is controlled by macOS system APIs. No hex values are set by
the app. This is the correct approach for menu bar apps that must adapt to dark and
light menu bar backgrounds.

| Role | Mechanism | Usage |
|------|-----------|-------|
| Menu bar icon — idle | `NSImage(systemSymbolName: "character.cursor.ibeam")` with `isTemplate = true` | Default state; macOS inverts automatically for dark/light menu bar |
| Menu bar icon — active | `NSImage(systemSymbolName: "pencil.circle.fill")` with `isTemplate = true` | Active vim session state; same dark/light adaptation |
| Menu item text | System-controlled (NSMenu renders at OS level) | All menu items |
| Disabled item text | System-controlled via `isEnabled = false` | Vim path info item, "custom path invalid" warning |

**Critical implementation requirement (from RESEARCH.md Pitfall 1):** `NSImage(systemSymbolName:)`
returns an image with `isTemplate = false` by default. The `isTemplate = true` assignment
MUST be made explicitly after every `NSImage(systemSymbolName:)` call that produces an
image assigned to `statusItem.button.image`. Failure to do this causes the icon to render
as a full-color image that does not invert on dark menu bars.

Accent reserved for: not applicable — no accent color is set; system handles all
visual state.

## SF Symbol Contract

These are the only two SF Symbols used in Phase 6 for the menu bar icon.

| State | Symbol Name | Accessibility Description | isTemplate |
|-------|-------------|--------------------------|------------|
| Idle (no vim session) | `character.cursor.ibeam` | `"AnyVim"` | `true` |
| Active (vim session open) | `pencil.circle.fill` | `"AnyVim — editing"` | `true` |

**Symbol rationale (Claude's discretion from CONTEXT.md):** `pencil.circle.fill` is chosen
from the pencil/edit family because:
- It has a filled variant that reads as distinctly "busy/active" vs the outline idle symbol
- The circle fill provides strong visual contrast in both light and dark menu bars in
  template mode
- It is available on macOS 13+ (target deployment target per CLAUDE.md)

Both symbols must be loaded fresh each time they are assigned to `button.image` — do not
cache the NSImage reference.

## Menu Item Contract

The vim path section is a new group added to `MenuBarController.buildMenu()`.
It is separated from existing items by NSMenuItem separators.

### Menu Structure (complete — showing only Phase 6 additions in context)

```
[Accessibility: Granted | Accessibility: Not Granted — Click to Enable]
[Input Monitoring: Granted | Input Monitoring: Not Granted — Click to Enable]
[Hotkey: Active | Hotkey: Inactive — Tap Unhealthy]
— separator (existing) —
[Vim: /opt/homebrew/bin/vim]          ← disabled info item (D-05)
    OR
[Vim: (custom path invalid)]          ← disabled info item, invalid custom path (D-06)
[Set Vim Path...]                     ← action item (D-04)
[Reset Vim Path]                      ← action item, visible only when custom path is set (D-04)
— separator —
[Launch at Login: Enabled | Launch at Login: Disabled]
— separator (existing) —
[Quit AnyVim]
```

### Menu Item Specifications

| Item | Title | Action | Enabled | Visible condition |
|------|-------|--------|---------|------------------|
| Vim path info | `"Vim: {resolvedPath}"` | none | false | Always |
| Vim path info (invalid) | `"Vim: (custom path invalid)"` | none | false | Custom path set but not executable |
| Set Vim Path... | `"Set Vim Path..."` | `#selector(setVimPath)` | true | Always |
| Reset Vim Path | `"Reset Vim Path"` | `#selector(resetVimPath)` | true | Only when `UserDefaults.standard.string(forKey: "customVimPath")` is non-nil and non-empty |

**Vim path info item display logic:**
- If custom path is set AND executable → `"Vim: {customPath}"`
- If custom path is set AND NOT executable → `"Vim: (custom path invalid)"`
- If no custom path → `"Vim: {ShellVimPathResolver result or 'vim not found'}"`

**Anti-pattern (RESEARCH.md Pitfall 3):** Do NOT call `ShellVimPathResolver.resolveVimPath()`
in `buildMenu()`. That method spawns `/bin/zsh -l -c "which vim"` which takes 100–300ms and
blocks the main thread on menu open. For the display item: read `UserDefaults.standard.string(forKey: "customVimPath")` directly. If no custom path is set, show a cached or static path string (or omit the shell resolution entirely — the display item's purpose is to show the custom path, not to re-resolve the PATH-based result on every menu open).

## NSOpenPanel Contract

The file picker opened by "Set Vim Path..." must be configured as follows.

| Property | Value | Reason |
|----------|-------|--------|
| `title` | `"Choose Vim Binary"` | Identifies purpose |
| `message` | `"Select the vim executable to use with AnyVim"` | Instruction |
| `prompt` | `"Select"` | Button label |
| `canChooseFiles` | `true` | Must select a file |
| `canChooseDirectories` | `false` | Directories are not executables |
| `allowsMultipleSelection` | `false` | Single binary only |
| `allowedContentTypes` | `[]` (empty — no UTI filter) | UTI system does not reliably classify Homebrew executables (RESEARCH.md Pitfall 4) |
| `directoryURL` | `URL(fileURLWithPath: "/usr/local/bin")` | Sensible default starting directory |

**Post-selection validation:** After `panel.runModal() == .OK`, call
`FileManager.default.isExecutableFile(atPath: url.path)`. If false, silently discard the
selection (do not save to UserDefaults). This matches D-06.

## Copywriting Contract

All copy for the new menu items and panels introduced in Phase 6.

| Element | Copy | Source |
|---------|------|--------|
| Vim path info item (valid path) | `"Vim: {path}"` | CONTEXT.md D-05 |
| Vim path info item (invalid custom path) | `"Vim: (custom path invalid)"` | CONTEXT.md D-06 |
| Set vim path menu item | `"Set Vim Path..."` | CONTEXT.md D-04 |
| Reset vim path menu item | `"Reset Vim Path"` | CONTEXT.md D-04 |
| NSOpenPanel title | `"Choose Vim Binary"` | Default (Claude's discretion) |
| NSOpenPanel message | `"Select the vim executable to use with AnyVim"` | Default (Claude's discretion) |
| NSOpenPanel button | `"Select"` | Default (Claude's discretion) |
| Icon accessibility — idle | `"AnyVim"` | RESEARCH.md (existing) |
| Icon accessibility — active | `"AnyVim — editing"` | RESEARCH.md Pattern 1 |

**Empty state:** Not applicable — the vim path info item always shows a value (either
the custom path, the system-resolved path, or "vim not found"). There is no empty state.

**Error state:** "custom path invalid" display in the disabled info item serves as the
error state for CONF-01. No alert or dialog is shown — the menu feedback is sufficient
per D-06 ("show '(custom path invalid)' so the user knows").

**Destructive actions:** "Reset Vim Path" clears the stored custom path. This is low-risk
(the user can re-set it) and does not require a confirmation dialog. It executes
immediately on click, consistent with the minimal-interruption menu bar daemon pattern
(CONTEXT.md D-02, CLAUDE.md §Menu Bar Integration).

## Interaction Contract

### Icon Swap (MENU-02)

| Trigger | Before trigger | After trigger |
|---------|---------------|---------------|
| `handleHotkeyTrigger()` starts | idle: `character.cursor.ibeam` | active: `pencil.circle.fill` |
| Session ends (any path) | active: `pencil.circle.fill` | idle: `character.cursor.ibeam` |

**Implementation contract (CONTEXT.md D-03):** Icon set to active BEFORE `captureText()`.
Icon restored to idle in the `defer` block alongside `isEditSessionActive = false`. The
`defer` block guarantees restoration on all exit paths: `:wq` save, `:q!` abort,
`captureText` failure, and any thrown error.

### Vim Path Set Flow (CONF-01)

1. User clicks "Set Vim Path..." in menu
2. NSOpenPanel opens modally (main thread, `@MainActor` — safe)
3. User selects a file and clicks "Select"
4. App calls `FileManager.default.isExecutableFile(atPath: selectedPath)`
5. If valid: `UserDefaults.standard.set(selectedPath, forKey: "customVimPath")` → `rebuildMenu()` called → menu now shows new path
6. If invalid: selection silently discarded — no dialog, no state change
7. On next hotkey trigger: `UserDefaultsVimPathResolver` uses the stored path

### Vim Path Reset Flow (CONF-01)

1. User clicks "Reset Vim Path" in menu (visible only when custom path is set)
2. `UserDefaults.standard.removeObject(forKey: "customVimPath")` executes immediately
3. `rebuildMenu()` called → "Reset Vim Path" item disappears, path display reverts to PATH-based resolution
4. On next hotkey trigger: `UserDefaultsVimPathResolver` falls back to `ShellVimPathResolver`
