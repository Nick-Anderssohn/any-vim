# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3: Accessibility Bridge and Clipboard.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | AppKit (NSMenu, NSMenuItem, NSAlert — system-provided) |
| Icon library | SF Symbols via NSImage(systemSymbolName:) |
| Font | System font (NSFont.systemFont — no custom font) |

**Rationale:** AnyVim is a pure AppKit/Swift macOS menu bar daemon. There is no web stack, no React, no CSS tokens file. shadcn is not applicable. All UI is system-native AppKit.

**Phase 3 UI scope:** Phase 3 builds `AccessibilityBridge` — a backend class with no user-visible window, panel, or alert of its own. The only UI surface touched in this phase is the existing NSMenu (for a new menu item showing bridge status) and the established NSAlert pattern for error conditions. No new visual components are introduced.

## Typography

Phase 3 introduces no new text surfaces. All text is rendered by AppKit system components using system fonts. The contract below reflects the established pattern from Phases 1-2.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Menu item label | 14pt (system default) | Regular (400) | 1.0 (single line, system-managed) |
| Alert message text | 13pt (system default) | Semibold (600) | 1.2 (system-managed) |
| Alert informative text | 11pt (system default) | Regular (400) | 1.5 (system-managed) |
| Menu section header | 11pt (system default) | Regular (400) | 1.0 (system-managed) |

**Source:** AppKit system font defaults. AnyVim does not set explicit NSFont values — all typography is inherited from the system. Established in Phase 1 NSAlert and NSMenuItem usage.

**Note on 1pt proximity (menu item label 14pt vs alert message text 13pt):** Both values are system-inherited defaults. AppKit assigns 14pt to NSMenuItem titles and 13pt to NSAlert message text. AnyVim does not set either value explicitly. The 1pt difference is intentional and unavoidable — these values cannot be changed without overriding AppKit defaults, which is not appropriate for a native menu bar utility.

## Color

Phase 3 introduces no new color values. All color is system-controlled via AppKit.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | NSColor.windowBackgroundColor | Menu background, alert background |
| Secondary (30%) | NSColor.controlBackgroundColor | Menu item backgrounds, separator regions |
| Accent (10%) | NSColor.controlAccentColor | Active/selected menu item highlight |
| Destructive | NSColor.systemRed | Error alert icon (system .warning alertStyle) |

Accent reserved for: selected NSMenuItem highlight state only. Not used for status indicators in Phase 3 — bridge status is text-only in the menu.

**Template color:** Status bar icon uses `isTemplate = true` (established in Phase 1 AppDelegate). This auto-adapts to dark/light menu bar. No explicit color set on the icon.

## Copywriting Contract

Phase 3 adds one new menu item and one new error alert condition to the existing NSMenu.

| Element | Copy |
|---------|------|
| Bridge status — idle | "Hotkey: Ready" (existing item label pattern from Phase 2) |
| Bridge status — capturing | "Capturing text..." |
| Bridge status — restoring | "Restoring text..." |
| Capture failure alert — message | "Text Capture Failed" |
| Capture failure alert — informative | "AnyVim could not read the text in the focused field. Make sure Accessibility permission is granted, then try again." |
| Capture failure alert — button | "Dismiss" |
| Empty field behavior | Silent — no alert. Vim opens with an empty file. No user-visible message. (Source: CONTEXT.md D-09) |
| Original app terminated mid-session | Silent — no alert. Focus restore is skipped. (Source: CONTEXT.md D-08) |
| Primary CTA | Not applicable — Phase 3 has no primary CTA. The trigger is the double-tap Control hotkey, established in Phase 2. |
| Destructive actions | None in Phase 3. Clipboard restore is automatic and non-destructive. Temp file deletion is automatic. |

**Implementation note — alert button:** The capture failure alert must use `alert.addButton(withTitle: "Dismiss")` explicitly. Do not rely on AppKit's auto-generated default button, which produces "OK" and cannot be overridden without an explicit `addButton` call.

**Copywriting rules (carried from Phase 1 pattern):**
- Alert message text: Title Case, noun phrase only (no verbs)
- Alert informative text: Sentence case, describes problem + what user should do next
- Menu items: Sentence case for status items; Title Case for action items

## Interaction States

Phase 3 introduces no interactive UI. The following state transitions happen silently (no visual feedback to the user) in Phase 3:

| Trigger | State | User sees |
|---------|-------|-----------|
| Double-tap Control fired | Capture begins | Nothing — no loading indicator in Phase 3 |
| Cmd+A / Cmd+C posted | Text captured | Nothing |
| Temp file written | Ready for vim | Nothing — Phase 4 opens vim |
| Capture fails (no Accessibility) | Error | NSAlert: "Text Capture Failed" |
| Empty field captured | vim opens empty | Nothing |
| Original app terminated | Restore skipped | Nothing |
| Clipboard restored | Cycle complete | Nothing |

**Note on visual feedback:** MENU-02 (menu bar icon animation during active session) is explicitly scoped to Phase 6 (ROADMAP.md). Phase 3 does not implement any menu bar icon state changes.

## Accessibility (macOS HIG Compliance)

Phase 3 has minimal new UI surface, but the following contracts apply:

- NSAlert buttons must use descriptive labels — use `alert.addButton(withTitle: "Dismiss")` explicitly; do not rely on AppKit's auto-generated "OK" default
- Error alert must use `.warning` alertStyle (amber icon) not `.critical` — capture failure is recoverable
- All NSMenuItem titles must be unique within their menu to support VoiceOver navigation
- No custom NSView with drawing code — system components handle accessibility automatically
