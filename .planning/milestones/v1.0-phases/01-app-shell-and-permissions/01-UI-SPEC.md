> no component library, and no shadcn. All design tokens map to AppKit/NSMenu

## Spacing Scale

AppKit menu and alert components use system-defined internal spacing.
The following tokens apply to any custom layout decisions within alerts
or future custom views.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps within custom views |
| sm | 8px | Compact element spacing in custom views |
| md | 16px | Default NSAlert informativeText paragraph spacing (system default) |
| lg | 24px | Section padding in any custom NSView |
| xl | 32px | Layout gaps in any custom NSView |
| 2xl | 48px | Major section breaks (not used in Phase 1) |
| 3xl | 64px | Page-level spacing (not used in Phase 1) |

Exceptions: NSAlert and NSMenu spacing is fully system-controlled. Do not
set custom insets on NSAlert buttons or NSMenuItem padding.

## Typography

AppKit system controls (NSAlert, NSMenu) use system-defined typography.
The following are binding constraints for any custom NSTextField or
NSText content added to alerts or views in this phase.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 13pt | Regular (400) | 1.5 | NSAlert informativeText, menu item secondary labels |
| Label | 13pt | Semibold (600) | 1.2 | NSAlert messageText (heading), menu item primary labels |
| Caption | 11pt | Regular (400) | 1.4 | Supplementary text, status labels below menu items (if any) |

Notes:
- NSAlert.messageText renders at approximately 13pt semibold (system default).
- NSAlert.informativeText renders at approximately 13pt regular (system default).
- NSMenuItem titles render at 13pt regular by default on macOS 13+.
- Do not override NSAlert or NSMenuItem font properties — use system defaults.
- If custom NSTextFields are added, use NSFont.systemFont(ofSize: 13) for body
  and NSFont.boldSystemFont(ofSize: 13) for labels.

## Color

Native AppKit controls use semantic system colors that automatically adapt
to light/dark mode and accessibility settings. Do not hardcode hex values
for any AppKit UI element in this phase.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | NSColor.windowBackgroundColor | Alert window background, menu background (system-managed) |
| Secondary (30%) | NSColor.controlBackgroundColor | Menu item backgrounds, separator areas (system-managed) |
| Accent (10%) | NSColor.controlAccentColor | "Open System Settings" button highlight, active toggle state |
| Destructive | NSColor.systemRed | Not used in Phase 1 — no destructive actions |
| Status: granted | NSColor.systemGreen | Permission status label when granted (if colored indicators are added) |
| Status: not granted | NSColor.systemOrange | Permission status label when not granted (if colored indicators are added) |

Accent reserved for: the "Open System Settings" primary button in NSAlert dialogs.
All other interactive elements (Quit, Later, Launch at Login toggle) use system
default AppKit styling.

Menu bar icon: use isTemplate = true on the NSImage. This makes the SF Symbol
automatically adapt to light/dark menu bar without any color code.

## Menu Structure

Exact menu item order and labels for Phase 1 dropdown (D-06):

```
[Section: Permission Status]
  "Accessibility: Granted"                     ← no action
  "Accessibility: Not Granted — Click to Enable" ← action: open System Settings
  "Input Monitoring: Granted"                  ← no action
  "Input Monitoring: Not Granted — Click to Enable" ← action: open System Settings

[Separator]

[Section: Configuration]
  "Launch at Login"    [checkmark if enabled]  ← action: toggle SMAppService

[Separator]

[Section: App]
  "Quit AnyVim"                                ← key equivalent: ⌘Q
```

Rules:
- Permission status items for granted permissions have no action (isEnabled = false or action = nil).
- Permission status items for missing permissions are clickable and open the correct System Settings pane.
- "Launch at Login" reads live state from SMAppService.mainApp.status on every menu open. Do not cache in UserDefaults.
- "Quit AnyVim" uses NSApplication.terminate(_:) as target/action with keyEquivalent "q".

## Interaction Contract

### First-Run Permission Flow (D-01, D-02)

1. On applicationDidFinishLaunching, check both permissions.
2. If Accessibility is missing: show NSAlert (Accessibility alert copy above). alertStyle = .warning.
3. If user clicks "Open System Settings": open the Accessibility pane URL. Dialog dismisses.
4. If user clicks "Later": dismiss without action.
5. Start permission polling (DistributedNotificationCenter + 3s Timer).
6. When Accessibility is detected as granted: post UNNotification (granted copy above). Then check Input Monitoring.
7. If Input Monitoring is missing: show NSAlert (Input Monitoring alert copy above). Same button behavior.
8. When Input Monitoring is detected as granted: post UNNotification.
9. Menu items update to "Granted" state within one poll cycle (≤3 seconds after grant).

### Menu Updates on Permission Change

- Menu is rebuilt (or status items updated) each time permissions change.
- Always re-query AXIsProcessTrusted() and CGPreflightListenEventAccess() live — never serve cached values.
- Granted items: action = nil, title uses "Granted" form.
- Missing items: action = openSettings handler, title uses "Not Granted — Click to Enable" form.

### Launch at Login Toggle (D-07, D-09)

- On first launch only: call SMAppService.mainApp.register() (D-07).
- Menu item state = .on when SMAppService.mainApp.status == .enabled (read live on each menu open).
- On toggle: call register() if currently disabled, unregister() if currently enabled.
- Rebuild menu after toggle to reflect updated checkmark.

### Re-Poll Interval

- 3 seconds (RESEARCH.md Pattern 2 — satisfies PERM-03 "within a few seconds").
- DistributedNotificationCenter fires immediately on Accessibility grant; Timer catches Input Monitoring changes.
