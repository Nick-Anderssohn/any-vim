**PermissionManager.swift** (per PERM-01, PERM-02, PERM-03, RESEARCH Pattern 2):

Define a protocol for testability:
```swift
protocol PermissionChecking {
    var isAccessibilityGranted: Bool { get }
    var isInputMonitoringGranted: Bool { get }
}
```

Implement `final class PermissionManager: PermissionChecking`:
- `var isAccessibilityGranted: Bool { AXIsProcessTrusted() }` — import ApplicationServices
- `var isInputMonitoringGranted: Bool { CGPreflightListenEventAccess() }` — import CoreGraphics
- Private properties: `private var pollTimer: Timer?`, `private var previousAccessibility = false`, `private var previousInputMonitoring = false`
- `func startMonitoring(onChange: @escaping (_ accessibilityChanged: Bool, _ inputMonitoringChanged: Bool) -> Void)`:
  1. Snapshot initial state into `previousAccessibility` and `previousInputMonitoring`
  2. Add DistributedNotificationCenter observer for `"com.apple.accessibility.api"` on `.main` queue — on fire, check both permissions, compute deltas, call onChange if changed, update previous state
  3. Create `Timer.scheduledTimer(withTimeInterval: 3.0, repeats: true)` — same logic: check both, compute deltas, call onChange if changed, update previous state
  4. Store timer in `pollTimer`
- `func stopMonitoring()`: remove DistributedNotificationCenter observer, invalidate timer
- `func openAccessibilitySettings()`: `NSWorkspace.shared.open(URL(string: "x-apple.systempreferences:com.apple.preference.security?Privacy_Accessibility")!)`
- `func openInputMonitoringSettings()`: `NSWorkspace.shared.open(URL(string: "x-apple.systempreferences:com.apple.preference.security?Privacy_ListenEvent")!)`

CRITICAL: Use plain `AXIsProcessTrusted()` for polling — NEVER use `AXIsProcessTrustedWithOptions` with kAXTrustedCheckOptionPrompt in the timer (Pitfall 3 from RESEARCH).

**LoginItemManager.swift** (per MENU-04, D-07, D-09, RESEARCH Pattern 3):

Define a protocol for testability:
```swift
protocol LoginItemManaging {
    var isEnabled: Bool { get }
    func enable()
    func disable()
    func enableIfFirstRun()
}
```

Implement `final class LoginItemManager: LoginItemManaging`:
- `var isEnabled: Bool { SMAppService.mainApp.status == .enabled }` — import ServiceManagement. Always read live from OS, NEVER cache in UserDefaults (Pitfall 4 from RESEARCH).
- `func enable()`: `try? SMAppService.mainApp.register()`
- `func disable()`: `try? SMAppService.mainApp.unregister()`
- `func enableIfFirstRun()`: check `UserDefaults.standard.bool(forKey: "launchAtLoginConfigured")`. If false, set it to true and call `enable()`. Per D-07, launch at login is enabled by default on first run.

**NotificationManager.swift** (per D-04, RESEARCH UNUserNotificationCenter example):

```swift
import UserNotifications

final class NotificationManager {
    func requestAuthorization() {
        UNUserNotificationCenter.current().requestAuthorization(options: [.alert, .sound]) { _, _ in }
    }

    func notifyPermissionGranted(_ permissionName: String) {
        let content = UNMutableNotificationContent()
        content.title = "AnyVim"
        content.body = "\(permissionName) permission granted."
        let request = UNNotificationRequest(
            identifier: UUID().uuidString,
            content: content,
            trigger: nil
        )
        UNUserNotificationCenter.current().add(request)
    }
}
```

Exact notification copy from UI-SPEC Copywriting Contract:
- Title: "AnyVim"
- Accessibility body: "Accessibility permission granted."
- Input Monitoring body: "Input Monitoring permission granted."

**Update AppDelegate.swift** to wire all managers and implement the sequential alert flow (D-01, D-02, D-08):

Add instance properties:
```swift
private var permissionManager: PermissionManager!
private var loginItemManager: LoginItemManager!
private var notificationManager: NotificationManager!
private var menuBarController: MenuBarController!
```

In `applicationDidFinishLaunching`, after statusItem setup:
1. Create all managers
2. Call `notificationManager.requestAuthorization()` — must be called early (Pitfall 6 from RESEARCH)
3. Call `loginItemManager.enableIfFirstRun()` — per D-07, enable login item on first run
4. Create `menuBarController = MenuBarController(permissionManager: permissionManager, loginItemManager: loginItemManager)`
5. Assign `statusItem.menu = menuBarController.buildMenu()`
6. Run the sequential alert flow (must happen AFTER UI is set up, per RESEARCH anti-patterns):
   - If `!permissionManager.isAccessibilityGranted`: show accessibility alert (exact copy from UI-SPEC):
     - title: "Accessibility Permission Required"
     - body: "AnyVim needs Accessibility access to detect the focused text field and simulate keyboard shortcuts to capture and restore text. Grant access in System Settings, then return to AnyVim.\n\nAnyVim will also launch at login automatically. You can change this in the menu bar."
     - Note: The launch-at-login mention (D-08) is appended to the FIRST alert only
     - alertStyle: .warning
     - Buttons: "Open System Settings" (first), "Later" (second)
     - If user clicks "Open System Settings": call `permissionManager.openAccessibilitySettings()`
7. Start permission monitoring with onChange callback:
   - Track which permissions were previously granted to detect transitions
   - When accessibility transitions from false to true: call `notificationManager.notifyPermissionGranted("Accessibility")`, then check input monitoring. If not granted, show input monitoring alert:
     - title: "Input Monitoring Permission Required"
     - body: "AnyVim needs Input Monitoring access to detect the double-tap Control key trigger globally. Grant access in System Settings, then return to AnyVim."
     - Same button pattern
   - When input monitoring transitions from false to true: call `notificationManager.notifyPermissionGranted("Input Monitoring")`
   - On any change: call `statusItem.menu = menuBarController.buildMenu()` to rebuild menu with live state
